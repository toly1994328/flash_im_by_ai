# 关系型数据库选型调研

## 背景

当前 flash-im 后端认证系统的用户数据（`users`、`sms_codes`、`next_id`）全部存储在内存 `HashMap` 中，服务重启即丢失。需要引入关系型数据库实现持久化。

技术栈：Rust + Axum + Tokio（全异步），需要考虑 Rust 生态中的驱动和 ORM 支持。

---

## 候选数据库对比

| 维度 | PostgreSQL | MySQL | SQLite | MariaDB |
|------|-----------|-------|--------|---------|
| 类型 | 客户端/服务器 | 客户端/服务器 | 嵌入式 | 客户端/服务器 |
| 协议 | 开源 (PostgreSQL License) | 开源 (GPL) / 商业双授权 | 公共领域 | 开源 (GPL) |
| 并发模型 | MVCC，读写不互斥 | InnoDB 行锁，高并发下锁竞争较多 | 写锁全库（WAL 模式可并发读） | 与 MySQL 类似，线程池优化更好 |
| JSON 支持 | 原生 `jsonb`，可索引、可查询 | `JSON` 类型，5.7+ 支持 | 3.38+ 有 JSON 函数，功能有限 | 与 MySQL 基本一致 |
| 全文搜索 | 内置 `tsvector/tsquery` | InnoDB 全文索引 | FTS5 扩展 | 与 MySQL 一致 |
| 扩展性 | 极强（自定义类型、函数、扩展插件） | 一般 | 无 | 一般 |
| 适合数据量 | TB 级 | TB 级 | GB 级（建议 < 几十 GB） | TB 级 |

### 1. PostgreSQL

**功能**
- 数据类型最丰富：数组、`jsonb`、`hstore`、范围类型、几何类型等
- 支持 CTE、窗口函数、物化视图、分区表
- 强事务支持，完整 ACID
- 内置发布/订阅（`LISTEN/NOTIFY`），对 IM 场景有天然亲和力

**性能**
- MVCC 实现成熟，读写并发性能优秀
- 复杂查询优化器强大
- 连接开销较高，生产环境建议配合连接池（PgBouncer 或应用层连接池）

**Rust 生态**
- `sqlx`：纯 Rust 异步驱动，编译期 SQL 校验，与 Tokio 完美集成，**一等公民支持**
- `tokio-postgres`：底层异步驱动
- `diesel`：同步 ORM，成熟但不适合纯异步架构
- `sea-orm`：基于 `sqlx` 的异步 ORM

**适用场景**
- 数据模型会逐步复杂化的项目（用户、好友关系、群组、消息记录）
- 需要 JSON 存储灵活字段（用户扩展信息、消息元数据）
- 中大规模部署

### 2. MySQL

**功能**
- 功能完善，覆盖绝大多数 Web 应用需求
- 8.0 后支持窗口函数、CTE、JSON 增强
- 主从复制成熟

**性能**
- 简单读写场景性能优秀
- 高并发写入时 InnoDB 行锁可能成为瓶颈
- 连接模型为线程模型，连接数上限相对较低

**Rust 生态**
- `sqlx` 支持 MySQL，但社区关注度和测试覆盖不如 PostgreSQL
- `mysql_async`：异步驱动
- `diesel` 支持 MySQL

**适用场景**
- 团队已有 MySQL 运维经验
- 读多写少的典型 Web 应用

### 3. SQLite

**功能**
- 零配置，无需独立进程，数据库就是一个文件
- SQL 功能子集，够用但不丰富
- 不支持并发写入（同一时刻只有一个写者）

**性能**
- 单线程写入，不适合高并发写场景
- 读性能优秀（尤其本地访问无网络开销）
- WAL 模式下可并发读

**Rust 生态**
- `sqlx` 支持 SQLite
- `rusqlite`：同步绑定，成熟稳定

**适用场景**
- 开发/测试环境
- 单机小规模部署、嵌入式场景
- **不适合** IM 这种多用户并发写入场景的生产环境

### 4. MariaDB

**功能**
- MySQL 分支，高度兼容 MySQL 语法
- 部分特性领先 MySQL（如更早支持窗口函数、虚拟列）
- Aria 存储引擎、ColumnStore 等差异化特性

**性能**
- 线程池优化优于 MySQL 社区版
- 整体与 MySQL 接近

**Rust 生态**
- 通常复用 MySQL 驱动（协议兼容）
- 专属特性可能缺少驱动支持

**适用场景**
- 想用 MySQL 但希望完全开源（无 Oracle 商业授权顾虑）
- 与 MySQL 场景基本一致

## 建议

**推荐 PostgreSQL**，理由：

1. **Rust 生态最佳适配**：`sqlx` 对 PostgreSQL 的支持最完善，编译期 SQL 检查、类型映射、异步支持都是一等公民级别，与你当前的 Axum + Tokio 异步架构无缝衔接。

2. **IM 场景天然契合**：
   - `LISTEN/NOTIFY` 可用于轻量级实时事件推送
   - `jsonb` 适合存储消息体、用户扩展字段等半结构化数据
   - MVCC 并发模型适合 IM 的高频读写混合负载

3. **可演进性强**：随着 flash-im 功能扩展（好友关系、群组、消息历史、已读状态等），PostgreSQL 的高级特性（数组、递归 CTE、物化视图）能持续支撑需求增长，不会成为瓶颈。

**推荐技术组合**：
```
PostgreSQL + sqlx + Tokio 连接池（sqlx 内置）
```

`Cargo.toml` 新增依赖参考：
```toml
sqlx = { version = "0.8", features = ["runtime-tokio", "tls-rustls", "postgres", "chrono", "json"] }
```
