# 登录状态管理演化报告

## 背景

Auth 模块的登录页需要管理表单交互状态（登录模式切换、验证码发送、加载状态等）。在实现过程中，经历了三次架构演化，每次都是在发现问题后及时纠正。

---

## 第一阶段：Sealed Class + LoginCubit

### 方案

使用 `flutter_bloc` 的 Cubit 模式，`LoginState` 为 sealed class，每种状态一个子类：

```dart
sealed class LoginState {}
class LoginInitial extends LoginState { final bool isSmsMode; }
class SmsSending extends LoginState {}
class SmsSent extends LoginState { final String code; final bool isSmsMode; }
class LoginLoading extends LoginState {}
class LoginSuccess extends LoginState { final LoginResult loginResult; final User user; }
class LoginFailure extends LoginState { final String message; final bool isSmsMode; }
```

### 问题

1. **状态维度混淆**：`isSmsMode`（UI 模式）是持久的，`SmsSending`/`LoginLoading`（操作状态）是瞬时的。两个维度被拆成不同的子类，每次状态切换都会丢失另一维度的信息。
2. **信息丢失**：`SmsSending` 和 `LoginLoading` 没有 `isSmsMode`，emit 后 UI 不知道当前模式。发完验证码（`SmsSent`）点登录变成 `LoginLoading`，验证码信息丢了。
3. **扩展性差**：加新登录方式需要在每个子类里都带上模式标识，子类数量膨胀。

### 反思

Sealed class 适合状态之间互斥且信息不重叠的场景。登录页的状态有两个独立维度（模式 + 操作），不适合用互斥子类建模。

---

## 第二阶段：去掉 Cubit，StatefulWidget + Mixin

### 思考

登录页的状态是页面级的，生命周期随页面销毁，不需要跨组件共享。它本质上就是"填表单 → 调接口 → 拿 token"，用状态管理框架是过度设计。

用 mixin 把状态数据和业务逻辑从 State 类里抽出来，LoginPage 的 State 只管 UI 构建，mixin 管数据和方法。职责分离，但不引入额外框架。

### 方案

```dart
mixin LoginMixin on State<LoginPage> {
  bool isSmsMode = true;
  bool isLoading = false;
  bool agreed = false;
  int countdown = 0;
  final phoneCtrl = TextEditingController();
  final credentialCtrl = TextEditingController();
  // sendSms(), login(), toggleMode()...
}

class _LoginPageState extends State<LoginPage> with LoginMixin {
  @override
  Widget build(BuildContext context) { /* 纯 UI */ }
}
```

### 改进

- 职责分离：逻辑在 mixin，视图在 State
- 无框架开销：不需要 BlocProvider、BlocConsumer
- 生命周期自然：页面销毁时 mixin 的 dispose 自动清理

### 遗留问题

所有登录方式的状态（手机号、验证码、密码、倒计时）都混在一个 mixin 里。切换模式时共用一个 `credentialCtrl`，状态互相干扰。如果加邮箱登录，mixin 会越来越臃肿。

而且仔细审视会发现，`phoneCtrl` 也不是真正共享的——验证码模式的"手机号"和密码模式的"账号"语义不同，后者可以是手机号/用户名/ID/邮箱。各登录方式有各自独立的输入控件和校验逻辑，不应该混在一起。

## 第三阶段：Mixin + 策略对象 + 抽象基类

### 思考

不同登录方式有各自的输入控件和校验逻辑，它们不是继承关系，而是组合关系——LoginMixin 同时持有多个策略对象，按需激活。切换模式只是换个指针，各策略的状态互不丢失。

同时，"怎么登录"这件事应该留在策略内部。如果 `login()` 写死在 Mixin 里，每种登录方式的调用逻辑都耦合在一起，新增方式要改 Mixin。更关键的是，三方登录（微信/Google）的流程完全不同——不是"填表单 → 调接口"，而是"OAuth 回调 → 拿 token"，写死在 Mixin 里根本塞不进去。

所以策略需要一个轻量的抽象基类，只约束共性：能校验、能登录、能销毁。`login()` 下沉到策略自身，Mixin 只做调度。

### 方案

抽象基类 `LoginStrategy`：

```dart
abstract class LoginStrategy {
  bool get isValid;
  Future<LoginResultData> login(AuthRepository repo);
  void dispose();
}
```

- `isValid` — 当前输入是否满足登录条件
- `login(repo)` — 执行登录，返回统一结果 `({LoginResult loginResult, User user})`
- `dispose()` — 释放资源

不强制 `sendSms()`、`countdown` 等特有能力——这些是具体策略的私有行为，不属于公共契约。

```
LoginStrategy（抽象基类）
├── isValid
├── login(repo) → LoginResultData
├── dispose()

LoginMixin（持有 + 调度）
├── agreed、isLoading、mode
├── smsStrategy: SmsLoginStrategy
├── passwordStrategy: PasswordLoginStrategy
├── currentStrategy → 当前激活的策略
├── login() → currentStrategy.login(repo) → AuthCubit.login → go('/home')
├── toggleMode() → 切换 mode

SmsLoginStrategy extends LoginStrategy
├── phoneCtrl、codeCtrl、countdown
├── login(repo) → repo.login(phone, credential, 'sms')
├── sendSms()（特有）

PasswordLoginStrategy extends LoginStrategy
├── accountCtrl（手机号/用户名/邮箱）、passwordCtrl
├── login(repo) → repo.login(phone, credential, 'password')
```

LoginMode 使用 Dart 增强枚举，携带接口所需的 type 值：

```dart
enum LoginMode {
  sms('sms'),
  password('password');

  final String value;
  const LoginMode(this.value);
}
```

LoginMixin 的 `login()` 简化为：

```dart
Future<void> login() async {
  final result = await currentStrategy.login(widget.authRepository);
  context.read<AuthCubit>().login(...);
  context.go('/home');
}
```

### 关键决策

**为什么接口只有三个方法？**

各策略的能力差异很大——SmsLoginStrategy 有 `sendSms()` 和 `countdown`，PasswordLoginStrategy 没有。硬抽太厚的公共接口，会逼密码策略实现无意义的方法。只约束共性（login/isValid/dispose），特有能力留在具体类。

**为什么同时持有所有策略？**

切换模式时旧策略的状态保留。用户在验证码模式发了验证码，切到密码模式再切回来，倒计时还在、验证码还在。如果只持有当前策略，切换时就要销毁重建，状态丢失。

**输入控件为什么不共享？**

验证码模式的"手机号"和密码模式的"账号"语义不同。密码模式后续要支持用户名/ID/邮箱 + 密码登录，账号输入是通用标识符。各自持有 controller，互不干扰。


### 扩展路径

- 加邮箱验证码登录：新建 `EmailLoginStrategy extends LoginStrategy`，LoginMixin 里注册，LoginPage 加一套输入 UI
- 加三方登录（微信/Google）：新建 `WechatLoginStrategy extends LoginStrategy`，OAuth 回调拿到 token 后走 `login(repo)` 统一返回
- 所有方式最终都通过 `LoginStrategy.login(repo)` 返回统一的 `LoginResultData`，汇入 `AuthCubit.login()`

---

## 演化总结

| 阶段 | 方案 | 问题 |
|------|------|------|
| 1 | Sealed Class + Cubit | 状态维度混淆，信息丢失，扩展性差 |
| 2 | StatefulWidget + 单 Mixin | 去掉框架开销，但所有登录方式混在一起 |
| 3 | Mixin + 策略对象 + 抽象基类 | 各登录方式独立，login() 下沉到策略，Mixin 只做调度 |

核心认知：
- 页面级状态不需要状态管理框架，`setState` 足够
- 多维度状态不适合 sealed class，组合优于继承
- 策略模式的公共接口应该足够轻——只约束共性（login/isValid/dispose），不强制特有能力
- 把"怎么登录"的知识留在策略内部，调度层只关心"登录"这个动作
- 目录结构应反映职责边界：data / logic / view 三层分离，logic 内部按功能域再分组
